Q: Set revisionHistoryLimit: 3 and explain how it affects your ability to roll back and how many old ReplicaSets are kept.
A: edit in kubectl edit deployments/web and set revisionHistoryLimit: to 3. it limits the number of old ReplicaSets retained for rollback. Only the three most recent previous revisions are kept. Older ReplicaSets are removed, so rollback is only possible to revisions that still exist.

Q: Use a label selector to list only the pods belonging to one Deployment, and explain the Deployment →
ReplicaSet → Pod selector relationship

A: A Deployment uses a label selector (for example, app=web) to manage its ReplicaSet. The ReplicaSet uses the same selector to create and manage Pods with matching labels, forming the relationship: Deployment → ReplicaSet → Pods.
kubectl get pods -l app=web

Q: Write a Deployment manifest for httpd:2.4 with 2 replicas, a named container port, and a nodeSelector; explain why it stays Pending if no node matches.
A:
apiVersion: apps/v1
kind: Deployment
metadata:
  name: httpd-deployment
spec:
  replicas: 2
  selector:
    matchLabels:
      app: httpd
  template:
    metadata:
      labels:
        app: httpd
    spec:
      nodeSelector:
        disktype: ssd
      containers:
      - name: httpd
        image: httpd:2.4
        ports:
        - name: http
          containerPort: 80

If no node has the label disktype=ssd, the Pods will remain in the Pending state because the scheduler cannot find a node that matches the nodeSelector requirement.

Q: Create a bare Pod (no controller) running busybox that sleeps, then explain what happens when you delete it versus a Deployment-managed pod.
A:
apiVersion: v1
kind: Pod
metadata:
  name: busybox-sleep
spec:
  containers:
  - name: busybox
    image: busybox
    command: ["sleep", "3600"]
kubectl apply -f busybox-pod.yaml

A bare Pod is not managed by any controller, so when it is deleted, it is gone permanently and Kubernetes does not recreate it. A Deployment-managed Pod is owned by a ReplicaSet, so if you delete the Pod, the ReplicaSet automatically creates a replacement Pod to maintain the desired number of replicas.

Q: 17. Create a DaemonSet running a busybox agent and explain why exactly one pod runs per node.
A:
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: busybox-agent
spec:
  selector:
    matchLabels:
      app: busybox-agent
  template:
    metadata:
      labels:
        app: busybox-agent
    spec:
      containers:
      - name: busybox
        image: busybox
        command: ["sh", "-c", "while true; do sleep 30; done"]
A DaemonSet ensures that one Pod is scheduled on every eligible node in the cluster. When a new node is added, the DaemonSet automatically creates a Pod on that node, and when a node is removed, the corresponding Pod is deleted.

Q: Create a multi-container Pod where two containers share an emptyDir — one writes a file, the other reads it.
Prove it's shared.

A:
apiVersion: v1
kind: Pod
metadata:
  name: shared-volume-pod
spec:
  volumes:
  - name: shared-data
    emptyDir: {}

  containers:
  - name: writer
    image: docker.io/library/busybox:latest
    command:
    - sh
    - -c
    - |
      echo "Hello from writer container" > /shared/message.txt
      sleep 3600
    volumeMounts:
    - name: shared-data
      mountPath: /shared

  - name: reader
    image: docker.io/library/busybox:latest
    command:
    - sh
    - -c
    - sleep 3600
    volumeMounts:
    - name: shared-data
      mountPath: /shared
proof: kubectl exec shared-volume-pod -c writer -- cat /shared/message.txt

read s file from container: kubectl exec shared-volume-pod -c reader -- cat /shared/message.txt
